# AKS Troubleshooting Commands for WSL

This document consolidates commands to resolve `kubectl` and `curl` connection issues when interacting with an Azure Kubernetes Service (AKS) cluster (`DemoCluster`) in WSL (Ubuntu). Each command is formatted for easy copying, with its purpose and usage instructions.

## Commands

### 1. Check for kubeconfig file in WSL
- **Command**:
  ```bash
  ls -l ~/.kube/config
  ```
- **Purpose**: Checks if a kubeconfig file exists in the WSL home directory (`/home/zakirh/.kube/config`), which `kubectl` uses to connect to the Kubernetes cluster.
- **Usage**: Run to verify the presence of the kubeconfig file. If it outputs `No such file or directory`, the kubeconfig is missing, and you need to copy or generate it.

### 2. Create directory for kubeconfig in WSL
- **Command**:
  ```bash
  mkdir -p ~/.kube
  ```
- **Purpose**: Creates the `~/.kube` directory in WSL if it doesn't exist, preparing it to store the kubeconfig file.
- **Usage**: Run before copying the kubeconfig file to ensure the directory is available.

### 3. Copy kubeconfig from Windows to WSL
- **Command**:
  ```bash
  cp /mnt/c/Users/ZakirHussain/.kube/config ~/.kube/config
  ```
- **Purpose**: Copies the kubeconfig file from the Windows path (`C:\Users\ZakirHussain\.kube\config`) to the WSL home directory (`~/.kube/config`) so `kubectl` in WSL can use it.
- **Usage**: Use when the kubeconfig file exists in Windows (generated by `az aks get-credentials`) but not in WSL. Verify with `ls -l ~/.kube/config`.

### 4. Set KUBECONFIG environment variable
- **Command**:
  ```bash
  export KUBECONFIG=/mnt/c/Users/ZakirHussain/.kube/config
  ```
- **Purpose**: Configures `kubectl` to use the kubeconfig file located in the Windows path without copying it to WSL.
- **Usage**: Run to point `kubectl` to the Windows kubeconfig file. Useful if copying the file is not preferred or fails due to permissions.

### 5. Make KUBECONFIG persistent
- **Command**:
  ```bash
  echo 'export KUBECONFIG=/mnt/c/Users/ZakirHussain/.kube/config' >> ~/.bashrc && source ~/.bashrc
  ```
- **Purpose**: Adds the `KUBECONFIG` environment variable to `~/.bashrc` to persist across WSL sessions and reloads the shell configuration.
- **Usage**: Run to ensure `kubectl` always uses the correct kubeconfig file in future WSL sessions. Replace `~/.bashrc` with `~/.zshrc` if using Zsh.

### 6. View kubeconfig contents
- **Command**:
  ```bash
  cat ~/.kube/config
  ```
- **Purpose**: Displays the contents of the kubeconfig file to verify it contains the correct AKS cluster details (e.g., server URL, context, and user credentials).
- **Usage**: Use to inspect the `clusters`, `contexts`, and `users` sections. Ensure the `server` field points to an Azure endpoint (not `localhost`).

### 7. List available kubectl contexts
- **Command**:
  ```bash
  kubectl config get-contexts
  ```
- **Purpose**: Lists all contexts in the kubeconfig file, showing which is currently active (marked with `*`).
- **Usage**: Run to verify that the `DemoCluster` context exists and is active. Use to identify the context to switch to if needed.

### 8. Set kubectl context to DemoCluster
- **Command**:
  ```bash
  kubectl config use-context DemoCluster
  ```
- **Purpose**: Sets the active context to `DemoCluster` so `kubectl` connects to the correct AKS cluster.
- **Usage**: Run if `kubectl config get-contexts` shows that `DemoCluster` is not the current context.

### 9. Test AKS cluster connectivity
- **Command**:
  ```bash
  kubectl get nodes
  ```
- **Purpose**: Tests connectivity to the AKS cluster by listing all nodes in the cluster.
- **Usage**: Run to confirm that `kubectl` can communicate with the AKS API server. Successful output lists node names (e.g., `aks-extended-12504507-vmss000003`).

### 10. List deployments across all namespaces
- **Command**:
  ```bash
  kubectl get deployments --all-namespaces
  ```
- **Purpose**: Retrieves a list of all deployments in the AKS cluster across all namespaces.
- **Usage**: Run to verify cluster access and list deployed applications. This was the original command that failed due to the `localhost:8080` error.

### 11. Start kubectl proxy
- **Command**:
  ```bash
  kubectl proxy --port=8001 &
  ```
- **Purpose**: Starts a local proxy server on `localhost:8001` that forwards requests to the AKS cluster's API server.
- **Usage**: Run in the background (with `&`) before using `curl` to query the Kubernetes API locally. Required for the `curl` command targeting `localhost:8001`.

### 12. Query node configuration with curl
- **Command**:
  ```bash
  curl -sSL "http://localhost:8001/api/v1/nodes/aks-extended-12504507-vmss000003/proxy/configz" | grep -i kubeReserved
  ```
- **Purpose**: Queries the configuration of a specific node (`aks-extended-12504507-vmss000003`) via the Kubernetes API to check `kubeReserved` settings.
- **Usage**: Run after starting `kubectl proxy` to access node configuration locally. Filters output for `kubeReserved` (resource reservations for Kubernetes components).

### 13. Describe node with kubectl
- **Command**:
  ```bash
  kubectl describe node aks-extended-12504507-vmss000003
  ```
- **Purpose**: Provides detailed information about a specific node, including resource allocations and annotations like `kubeReserved`.
- **Usage**: Run as an alternative to `curl` to inspect node details. Use `| grep -i kubeReserved` to filter for specific settings if needed.

### 14. Query kubeReserved with kubectl
- **Command**:
  ```bash
  kubectl get node aks-extended-12504507-vmss000003 -o jsonpath='{.metadata.annotations.kubeReserved}'
  ```
- **Purpose**: Extracts the `kubeReserved` annotation (if present) from a specific node's metadata.
- **Usage**: Run to directly retrieve `kubeReserved` settings without parsing the full node description. May return empty if not explicitly set.

### 15. Check Azure CLI login status
- **Command**:
  ```bash
  az account show
  ```
- **Purpose**: Displays the current Azure CLI authentication status and active subscription.
- **Usage**: Run to verify that you are logged into Azure with the correct account and subscription to access the AKS cluster.

### 16. Regenerate kubeconfig with Azure CLI
- **Command**:
  ```bash
  az aks get-credentials --resource-group n00482_westeurope --name DemoCluster --overwrite-existing
  ```
- **Purpose**: Fetches or updates the kubeconfig file for the AKS cluster (`DemoCluster`) in the specified resource group (`n00482_westeurope`).
- **Usage**: Run if the kubeconfig file is missing or stale. This updates `C:\Users\ZakirHussain\.kube\config` (Windows path), which must then be copied to WSL.

### 17. Install Azure CLI in WSL
- **Command**:
  ```bash
  curl -sL https://aka.ms/InstallAzureCLIDeb | sudo bash
  ```
- **Purpose**: Installs the Azure CLI in WSL (Ubuntu) to run `az` commands locally.
- **Usage**: Run if Azure CLI is not installed in WSL and you need to execute `az aks get-credentials` within WSL.

### 18. Log in to Azure CLI
- **Command**:
  ```bash
  az login
  ```
- **Purpose**: Authenticates the Azure CLI with your Azure account.
- **Usage**: Run after installing Azure CLI or if `az account show` indicates you are not logged in.

### 19. Check for local Kubernetes processes
- **Command**:
  ```bash
  ps aux | grep kube
  ```
- **Purpose**: Lists running processes related to Kubernetes to check for local clusters (e.g., minikube, kind) that might interfere with AKS connectivity.
- **Usage**: Run to identify and stop local Kubernetes services if they are causing `localhost` connection attempts.

### 20. Stop minikube
- **Command**:
  ```bash
  minikube stop
  ```
- **Purpose**: Stops a locally running minikube cluster that might be listening on `localhost:8080` or `localhost:8001`.
- **Usage**: Run if `ps aux | grep kube` shows a minikube process interfering with AKS commands.

### 21. Delete kind cluster
- **Command**:
  ```bash
  kind delete cluster
  ```
- **Purpose**: Deletes a locally running kind cluster that might be interfering with AKS connectivity.
- **Usage**: Run if `ps aux | grep kube` shows a kind process causing `localhost` connection issues.

### 22. Clear kubectl cache
- **Command**:
  ```bash
  rm -rf ~/.kube/cache
  ```
- **Purpose**: Removes cached `kubectl` data that might cause connection errors due to stale information.
- **Usage**: Run if `kubectl` commands fail despite a valid kubeconfig, to reset cached API discovery data.

### 23. Test network connectivity
- **Command**:
  ```bash
  ping google.com
  ```
- **Purpose**: Verifies that WSL has network connectivity to external services.
- **Usage**: Run to ensure WSL can reach the internet, which is necessary for connecting to the AKS API server.

### 24. Update WSL DNS settings
- **Command**:
  ```bash
  sudo nano /etc/resolv.conf
  ```
  - Add: `nameserver 8.8.8.8`
- **Purpose**: Updates WSL's DNS configuration to use a public DNS server (Google's 8.8.8.8) to fix network resolution issues.
- **Usage**: Run if `ping google.com` fails. Edit `/etc/resolv.conf`, add the DNS line, save, and exit. Then restart WSL.

### 25. Restart WSL
- **Command**:
  ```powershell
  wsl --shutdown
  ```
- **Purpose**: Shuts down the WSL environment to apply network or configuration changes.
- **Usage**: Run from a Windows PowerShell or Command Prompt after updating `/etc/resolv.conf` or making other system changes. Reopen the WSL terminal afterward.

### 26. Check Windows kubeconfig file
- **Command**:
  ```powershell
  dir C:\Users\ZakirHussain\.kube\config
  ```
- **Purpose**: Verifies that the kubeconfig file exists in the Windows path.
- **Usage**: Run in Windows PowerShell or Command Prompt to confirm the file is present before copying to WSL. If missing, regenerate with `az aks get-credentials`.

### 27. Install kubectl in WSL
- **Command**:
  ```bash
  curl -LO "https://dl.k8s.io/release/v1.33.1/bin/linux/amd64/kubectl"
  chmod +x kubectl
  sudo mv kubectl /usr/local/bin/
  ```
- **Purpose**: Installs a stable version of `kubectl` (v1.33.1) in WSL to ensure compatibility and reliability.
- **Usage**: Run if the automatically downloaded `kubectl` version causes issues or to ensure a specific version is used.

### 28. Describe node capacity and allocatable resources
- **Command**:
  ```bash
  kubectl describe node aks-extended-12504507-vmss000003 | grep -e Capacity -A6 -e Allocatable -A6
  ```
- **Purpose**: Displays the `Capacity` and `Allocatable` resource sections for a specific node (`aks-extended-12504507-vmss000003`), including details like CPU, memory, and pod limits.
- **Usage**: Run to inspect the total and available resources on the node. The `grep` filters the output to show the `Capacity` section (total resources) and `Allocatable` section (resources available for scheduling), each with the next 6 lines for context.

## Notes
- **Order of execution**: Start with commands 1–3 to set up the kubeconfig, then verify with 6–9. Use 11–14 and 28 to address node-specific queries (`kubeReserved`, `Capacity`, `Allocatable`). Use 15–18 if Azure CLI setup is needed. Use 19–22 to rule out local cluster interference, and 23–25 for network issues.
- **Environment**: These commands assume WSL (Ubuntu) with access to a Windows kubeconfig file. Adjust paths if your Windows username differs.
- **AKS specifics**: The commands target the `DemoCluster` AKS cluster in the `n00482_westeurope` resource group. Modify names as needed for other clusters.
- **Troubleshooting**: If commands fail, check outputs and provide them for further assistance (e.g., `kubectl config view --minify`, `kubectl get nodes`).

This Markdown serves as a reference for troubleshooting AKS connectivity issues in WSL, ensuring `kubectl` and `curl` commands work correctly with your cluster.
